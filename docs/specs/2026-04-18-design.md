# Design

_Working name: `new123`. Real name TBD._

## What this is

An autonomous-experiment runner that builds a compounding theory of the
problem it's working on, instead of forgetting everything between runs.

An agent drives the loop. The agent proposes a change, the runner executes it
in isolation, the result gets recorded, and — periodically — the runner pauses
and asks the agent to sit down and update a persistent wiki: what do we now
believe, what did we rule out, what's still open. The next cycle's ideas come
out of the wiki, not out of a flat log.

Two gears. Explore is fast. Consolidate is slow. The wiki is the thing that
carries.

## Why not ACE or autoresearch

Both treat memory as a flat log and trust the metric blindly. That works until
experiment 50, at which point the agent starts re-proposing things it already
tried and the metric has drifted from what anyone actually cares about. Also,
both have real bugs — shell injection via yaml fields, TSV corruption, no budget
caps, no concurrency safety, no worktree isolation. Addressed below.

## Non-goals

- Not a cloud orchestrator. One machine, one project.
- Not a hyperparameter optimizer. No Bayesian opt, no bandits. The agent thinks.
- Not tied to any LLM vendor. Claude Code, Codex, plain bash — all fine.
- Not a metrics platform. Plain files on disk.
- Not multi-user. One user, one SSH session, one daemon.

## The two processes

- `newd` — long-lived daemon. Owns the experiment lifecycle: queue, worker
  pool, budget ledger, worktree mgmt, metric parsing, SQLite state, socket
  listener. Supervisor pattern — parent restarts child on crash.
- `new` — thin client. Each invocation is one RPC, then exit.

They talk over a Unix domain socket at `.new/sock`. No TCP by default. No port
to conflict with. `--listen tcp://host:port` exists for remote agents but is
opt-in and off for personal use.

## The three stores

1. `.new/state.db` — SQLite, WAL mode. Queue, runs, budget, session state.
   Authoritative. Machine-read.
2. `.new/logs/` — per-experiment stdout/stderr, one file per run, gzipped after
   N days. Human-read on demand.
3. `wiki/` — markdown tree. Agent read-write, human read. Git-tracked.

## Wiki layout (the palace)

MemPalace's wings/rooms/drawers mapped onto experiment memory:

- `theses/` — wings. High-level beliefs the work is converging on. Each one a
  page, dated, with current status (`forming`, `supported`, `contradicted`,
  `abandoned`). Titled like a claim: `wider-nets-beat-deeper-nets.md`.
- `topics/` — rooms. Thematic clusters. `learning-rate.md`,
  `regularization.md`. A topic page synthesizes what the experiments in that
  cluster tell us. Cites by experiment filename.
- `experiments/` — drawers. Per-experiment verbatim pages. Hypothesis, diff
  link, metric, holdout metric, verdict, backlinks to the theses/topics that
  were touched by this result.
- `index.md` — content-oriented catalog. Every page listed, one-line summary.
  Regenerated by `new lint`.
- `log.md` — chronological append-only. One line per event (`ingest`, `query`,
  `consolidate`, `lint`). Parseable by `awk`.
- `schema.md` — Karpathy-gist-style "how is this palace organized". Stable.

Agent writes all of it. Daemon only updates `index.md` and `log.md` as a
sanity layer — if the agent forgets, the daemon fills in.

## Data model (SQLite)

```sql
-- pending experiments
CREATE TABLE queue (
  id          INTEGER PRIMARY KEY AUTOINCREMENT,
  spec        TEXT NOT NULL,         -- json: hypothesis, diff_ref, notes
  priority    INTEGER NOT NULL DEFAULT 0,
  queued_at   REAL NOT NULL,
  claimed_by  TEXT,                  -- worker id
  claimed_at  REAL
);

-- completed experiments
CREATE TABLE runs (
  id              INTEGER PRIMARY KEY AUTOINCREMENT,
  exp_num         INTEGER NOT NULL UNIQUE,       -- 1, 2, 3, ...
  commit_sha      TEXT NOT NULL,
  metric          REAL,
  metric_holdout  REAL,
  status          TEXT NOT NULL CHECK(status IN ('keep','discard','crash','skip')),
  duration_s      REAL NOT NULL,
  started_at      REAL NOT NULL,
  ended_at        REAL NOT NULL,
  hypothesis      TEXT,
  verdict         TEXT,
  description     TEXT,
  log_path        TEXT,
  wiki_refs       TEXT                           -- json list
);

-- budget ledger, one row per cap
CREATE TABLE budget (
  key         TEXT PRIMARY KEY,   -- 'experiments' | 'wallclock' | 'cost'
  used        REAL NOT NULL,
  cap         REAL NOT NULL,
  updated_at  REAL NOT NULL
);

-- k/v for session state
CREATE TABLE kv (
  k  TEXT PRIMARY KEY,
  v  TEXT NOT NULL
);

-- consolidation passes
CREATE TABLE consolidations (
  id              INTEGER PRIMARY KEY AUTOINCREMENT,
  triggered_by    TEXT CHECK(triggered_by IN ('count','stall','manual')),
  started_at      REAL NOT NULL,
  ended_at        REAL,
  pages_touched   INTEGER,
  notes           TEXT
);

-- baseline noise profile
CREATE TABLE baseline (
  n       INTEGER NOT NULL,
  mean    REAL NOT NULL,
  stddev  REAL NOT NULL,
  samples TEXT NOT NULL,     -- json list of metric values
  taken_at REAL NOT NULL
);
```

## RPC surface

Framing: length-prefixed msgspec frames on a SOCK_STREAM Unix socket. 4-byte
big-endian length, then payload. Every request has a type tag; server
dispatches.

Client verbs (= CLI subcommands of `new`):

| Verb          | Does                                                     |
|---------------|----------------------------------------------------------|
| `init`        | Scaffold `experiment.yaml`, `wiki/`, `.new/`. No daemon. |
| `up`          | Start `newd` in background. Idempotent.                  |
| `down`        | Graceful stop. SIGTERM, 10s, SIGKILL.                    |
| `status`      | Queue size, active runs, budget used, last 5 results.    |
| `baseline`    | Run baseline N times, persist mean/stddev.               |
| `run`         | Enqueue one experiment. Returns exp_num.                 |
| `record`      | Agent overrides a run's metric/status/verdict.           |
| `consolidate` | Trigger consolidation. Emits prompt on stdout.           |
| `lint`        | Wiki lint pass. Emits findings.                          |
| `resume`      | Restore queue from SQLite, unclaim stale claims.         |
| `logs`        | Tail a run's log, `--follow` for live.                   |
| `watch`       | Stream daemon events until Ctrl-C.                       |

Daemon-internal RPCs (not on CLI):

- `worker_claim` — worker pulls from queue
- `worker_done` — worker reports run result
- `budget_check` — atomic budget decrement before spawn
- `health` — supervisor liveness ping

## Two-gear trigger logic

Gear 2 (consolidation) fires on ANY of:

1. **Count.** `runs_since_last_consolidation >= consolidate_every` (default 20).
2. **Stall.** 5 consecutive `discard`, or stddev of last 10 metrics is below
   `improvement_threshold * 0.1` (metric has gone flat).
3. **Manual.** `new consolidate --force`.

When fired:

- Daemon sets `kv.consolidating='1'`, new claims from `queue` pause.
- Daemon assembles a consolidation prompt: recent N run summaries, current
  wiki snapshot (page tree + headings), stall diagnostics, plus a short
  instruction block.
- Prompt is written to stdout of the `new consolidate` call — the agent's
  context.
- Agent updates wiki files directly, commits, then calls
  `new consolidate --done --notes "..."`.
- Daemon records the pass in `consolidations`, clears the flag, resumes queue.

Budget enforcement does not pause — wall-clock keeps counting. But consolidation
has its own sub-budget (default 10 min) so an agent that hangs here can't burn
the whole session.

## Budget enforcement

`experiment.yaml`:

```yaml
run_command: "python train.py"     # split via shlex at load; stored as list
grep_pattern: "^val_loss:"
metric:
  name: "val_loss"
  direction: minimize              # or 'maximize'

budget:
  max_experiments: 500
  max_wallclock_hours: 8
  max_cost_usd: 0                  # 0 = disabled; if >0, agent reports cost

workers: 1                         # parallel experiment slots
consolidate_every: 20
consolidate_budget_minutes: 10     # cap on time spent in one gear-2 pass
improvement_threshold: 0.005       # minimum metric delta to call it a "real" improvement
session_tag: ""                    # blank → daemon asks agent to pick on 'new up'
```

Each cap has a row in `budget`. Before a worker spawns:

```
BEGIN IMMEDIATE;
  SELECT used, cap FROM budget WHERE key = ?;
  -- if used + projected > cap: abort, write halt reason, return
  UPDATE budget SET used = used + ?, updated_at = ? WHERE key = ?;
COMMIT;
```

Atomic. Two workers cannot both slip past the final experiment.

## Anti-Goodhart: held-out eval

Optional but strongly recommended. In `experiment.yaml`:

```yaml
holdout:
  run_command: "python eval_holdout.py"
  grep_pattern: "^holdout_metric:"
  every: 5                       # run every 5th kept experiment
```

When configured, every Nth `keep` decision triggers a holdout run before the
keep is finalized. If holdout metric diverges from primary metric (direction or
> threshold), the run is demoted to `discard` with verdict explaining the
divergence. The agent cannot see the holdout code. The wiki only gets the
aggregated divergence pattern.

## Baseline noise profile

`new baseline [--n 3]` runs the unmodified baseline N times, no code changes,
fresh worktrees each time. Computes mean, stddev. Saves to `baseline`.

On startup of the main loop, daemon compares `improvement_threshold` against
baseline stddev. If `stddev > improvement_threshold * 0.5`, daemon refuses to
start the loop and emits (example with threshold=0.1):

```
baseline stddev is 0.08; improvement threshold is 0.10.
your metric is noisier than half your target signal. either:
  - fix the rng seed in the experiment script
  - raise improvement_threshold in experiment.yaml
  - run 'new baseline --n 10' to get a tighter estimate
```

This catches the autoresearch/ACE failure mode where the agent "improves" the
metric by riding noise for a dozen cycles.

## Shell safety

- `run_command` and `holdout.run_command` are split at yaml load time via
  `shlex.split` into lists. The list is stored. Workers spawn with
  `subprocess.run(args=list, shell=False)`. No re-interpolation ever.
- `grep_pattern` is `re.compile`'d at yaml load. Invalid regex = yaml refused.
- Any yaml field used in a filename is validated against `[\w\-./]+` before
  touching the filesystem.
- UDS socket permissioned `0600`. Only the owning uid can connect.
- Worktrees are created under `.new/worktrees/<exp_num>-<short_sha>/`. Paths
  are absolute and validated to be under project root before any write.
- `pyproject.toml` pins every dep. `uv.lock` committed.

## Worktree isolation

Every experiment runs in its own `git worktree`. The agent edits files in
main checkout; when `new run` is called, daemon:

1. Creates worktree at `.new/worktrees/<exp_num>-<sha>/`.
2. Copies current agent-edited files into the worktree (or, if the agent
   commits first, checks out that commit).
3. Runs the experiment there.
4. On success: fast-forward the experiment branch to include the commit.
5. On crash/discard: remove worktree, leave no trace.

Main branch is never touched. All experiment commits are on
`new/<session-tag>` (e.g. `new/apr18`).

## Concurrency

Worker pool size from yaml field `workers` (default 1). For parallel runs,
each worker gets its own worktree and its own log file. `runs.exp_num` is
assigned from a counter row in `kv` (`k='next_exp_num'`), bumped inside the
same `BEGIN IMMEDIATE` transaction that claims the queue item. Two workers
can never receive the same `exp_num`.

Two `new watch` clients can connect simultaneously — each gets a fresh
event-stream from daemon. Reads are lock-free (WAL).

Two agents on two sessions can share one daemon, but they'll share the
budget. Different sessions should use different project directories.

## Error semantics

| Situation            | Outcome                                                   |
|----------------------|-----------------------------------------------------------|
| Worker crash         | run = `crash`, last 50 log lines emitted, worktree cleaned |
| Metric not found     | run = `crash`, with hint pointing at grep_pattern         |
| Worker timeout       | SIGTERM worker process group, 5s, SIGKILL. run = `crash`  |
| Daemon crash         | supervisor restarts, up to 5x, exponential backoff        |
| Stale PID file       | detect via /proc check, clean, continue                   |
| SQLite locked        | retry 3x with 50ms jitter, then raise                     |
| Yaml invalid         | `new up` refuses, prints msgspec validation error         |
| Budget exceeded      | queue drained, state=`halted`, agent notified on next call|
| Wiki path escape     | write refused, logged as a security event                 |

## Agent-facing contract (program.md, short version)

What the agent must do, in order:

1. `new status` — see if daemon is up, queue, budget.
2. If first run: `new init` then `new up` then `new baseline`.
3. Pick an idea. Write a one-sentence hypothesis.
4. Make the code change (in main checkout; worktree is on the daemon's side).
5. `git commit -m "..."` on the experiment branch.
6. `new run --hypothesis "..."` — daemon queues + runs.
7. Wait for result: `new logs --exp <N> --follow` or `new status`.
8. On result, write `wiki/experiments/<N>-<slug>.md` with hypothesis, diff
   summary, metric, verdict. Link to the topics/theses it touched; update
   those if meaningful.
9. If daemon emits a consolidation prompt (shows up on `new status` or piped
   via `new consolidate`), drop into gear 2: re-read recent experiments, update
   wiki synthesis, commit, call `new consolidate --done`.
10. Loop. Stop when `new status` reports `halted` or you're out of ideas.

The agent never writes SQLite directly. The agent never writes `results.tsv`.
The agent writes markdown (wiki) and Python (experiment code).

## Test strategy

- **Unit**: `schema`, `budget`, `wiki.path_validation`. Pure logic, fast.
- **Integration**: spawn a real `newd` on a temp dir. Exercise the full RPC
  surface from a real client. No mocking.
- **E2E**: `tests/test_e2e.py` runs `examples/toy-sort/` through five
  experiments, asserts the log, wiki pages, SQLite state, and worktree
  cleanup.
- **Property**: hypothesis-based test on JSONL round-trip (writing escaped
  content and reading it back, no corruption).
- **Security regressions**: test_shell_safety.py — every yaml field that
  could contain shell metachars gets a malicious value, and we assert that
  subprocess is called with argv, not a shell string.

## Out of scope for v1

- Multi-machine. One host.
- GPU scheduling. Experiment grabs what it grabs.
- Web UI. Terminal is the UI.
- Cost tracking via pricing APIs. If you want cost caps, report them yourself.
- Built-in hyperparameter search. The agent is the search.
- Windows. Linux and macOS only; uses `flock`, Unix sockets, SIGTERM.

## What makes this different from ACE, autoresearch, MemPalace, llm-wiki

- vs **ACE**: real code instead of 165 lines of prose, shell-safe, budget-enforced, concurrency-safe, worktree-isolated. Plus the wiki + two-gear loop, which ACE doesn't have.
- vs **autoresearch**: memory compounds instead of resetting. Held-out eval. Baseline noise check before the loop.
- vs **MemPalace**: domain-specific for experiments. Memory is not conversational text, it's hypothesis + diff + metric + verdict. Wiki structure reflects that.
- vs **llm-wiki**: the wiki isn't just a knowledge base — it's the agent's own memory of an active experiment loop. Ingest + query + lint are all present, plus the two-gear loop tying wiki updates to live experiment cadence.

## Open questions to resolve during implementation

1. Consolidation prompt format — does the daemon include raw log snippets
   or just metric curves? Will settle after first dogfood.
2. Default for `consolidate_every`. 20 is a guess. May adjust after one
   real session.
3. How strict is "commit required before run"? Currently strict. May loosen
   to "dirty tree allowed in --dev mode".
4. Wiki lint depth — cheap pass (orphans, broken links) vs deep pass
   (LLM-assisted contradiction detection). Start with cheap.
